Cocos 嵌入方案对比

Web UI 如何被 Cocos 预览页面的 iframe 加载？

方案 A：嵌入模式（URL 参数切换）
iframe URL
http://localhost:5173?embed=true&cwd=/path/to/cocos-project
工作原理：
  • 同一份代码，同一个构建产物
  • 通过 URL 参数 embed=true 激活嵌入模式
  • 嵌入模式下隐藏项目选择，锁定 cwd
  • 顶部 bar、会话列表等功能保持不变
代码改动
App.tsx 读取 URL 参数 → sessionStore 锁定 cwd → 条件渲染隐藏项目切换
方案 B：独立入口点（单独打包）
iframe URL
http://localhost:5173/cocos.html
工作原理：
  • Vite 多入口：index.html + cocos.html
  • Cocos 入口有独立的 App 组件（CocosApp）
  • 不包含项目选择逻辑，硬编码单项目模式
  • 共享 components/stores/hooks，只是组装不同
代码改动
新增 cocos.html + CocosApp.tsx → 独立路由树 → 复用底层组件

详细对比

维度 A — 嵌入模式 B — 独立入口
代码量 少，条件判断 ~50 行 中，新入口 + App ~150 行
维护成本 低，一份代码自动同步 中，两个入口需同步更新
产物体积 相同（全量加载） 可 tree-shake 掉项目选择代码
灵活性 受限于主 App 的结构 可自由定制 Cocos 特有 UI
Cocos 模板集成 改 iframe URL 加参数即可 改 iframe URL 指向 /cocos.html
未来扩展 如需更多差异化会越来越多 if/else 天然隔离，Cocos 可独立演进
A

嵌入模式（推荐）

最小改动，URL 参数控制。适合 Cocos 端和 Web 端功能高度一致的场景。你说的"其他功能和 web 保持一样"正好符合这个方案。

B

独立入口点

更干净的隔离，适合 Cocos 端将来需要大量差异化功能的场景。初期成本稍高但扩展性更好。
