服务器 GUI 方案对比

三种技术方案，各有权衡。点击选择你倾向的方案。

A

Electron 桌面应用（推荐）

方案：用 Electron 包装，GUI 内嵌启动 Fastify 服务器（同进程），系统托盘图标 + 管理窗口。

Pros

  • 复用现有 Node.js + TypeScript 技术栈，零学习成本
  • 系统托盘图标 — 不会被误关，最小化不占任务栏
  • 内嵌服务器 — 架构简单，一个进程搞定
  • 跨平台（Win/Mac/Linux），适合开源发布
  • 可以直接复用 web 端组件做管理界面
  • 成熟的打包工具（electron-builder）

Cons

  • 安装包体积较大（~80-150MB），内含 Chromium
  • 内存占用较高（~100-200MB 基线）
  • Electron 版本管理有一定维护成本
B

Tauri 轻量桌面应用

方案：Rust 做外壳 + WebView2 渲染 UI，通过 sidecar 机制管理 Node.js 服务器子进程。

Pros

  • 安装包极小（~5-15MB），用系统自带 WebView
  • 内存占用低（~30-50MB 基线）
  • 系统托盘支持好，原生感更强
  • 跨平台，适合开源

Cons

  • 需引入 Rust 工具链，团队技术栈扩展
  • Sidecar 管理 Node 子进程 — 架构复杂度高
  • 前端代码需适配 Tauri API（不能直接用 Node API）
  • Windows 上依赖 WebView2（Win10+ 一般已预装）
C

Node.js 原生托盘（node-systray / PowerShell）

方案：不用任何 GUI 框架，直接用 node-systray 或调 PowerShell 创建系统托盘图标，管理界面复用 Web UI。

Pros

  • 零额外依赖，安装包最小
  • 架构最简单 — 就是现有服务器 + 托盘图标
  • 管理界面直接复用 Web 端

Cons

  • node-systray 库成熟度一般，跨平台兼容性存疑
  • 没有独立窗口，所有管理都走浏览器
  • 开机自启、进程保护等功能需要自己实现
  • 作为开源项目，用户体验不够 polished

💡 我的推荐：方案 A — Electron

理由：项目是开源发布，需要良好的安装体验和跨平台支持。Electron 完美复用你现有的 Node.js + TypeScript + React 技术栈，不引入新语言。体积和内存的代价对桌面应用来说完全可以接受。而且 Fastify 服务器可以直接内嵌在 Electron 主进程中，架构最简单。
