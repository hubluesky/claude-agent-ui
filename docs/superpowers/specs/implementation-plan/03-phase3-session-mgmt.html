3.1 Session Resume（恢复历史会话交互）
复杂度: 中 改动文件: 5 SDK: options.resume（已部分使用）

问题

当前点击历史会话只能查看消息记录，无法继续对话。SDK 的 resume 选项已在 v1-session.ts 中使用，但仅用于同一 session 内的连续消息。缺少从侧边栏恢复历史会话的 UI 入口。

方案

在 SessionCard 上区分"查看历史"和"恢复交互"两种操作。

流程

  1. 点击"继续对话" → 发送 C2S_SendMessage 时带 resumeSessionId
  2. Server 端创建 V1QuerySession({ resumeSessionId })

  4. 点击"查看历史" → 仅加载消息，不创建活跃 session

3.3 Effort 自适应

3.4 Session Fork
复杂度: 中 改动文件: 5 SDK: forkSession(sessionId)

问题

用户想从某个历史节点"分叉"出新对话方向，保留之前的上下文但走不同路径。

UI 表现

Message hover — Fork button
assistant
我已经完成了登录功能的修复，主要改动包括 token 刷新逻辑和错误处理...
Sidebar — forked session appears
⚲ 修复登录 bug (fork)
just now · forked

流程

  1. hover assistant 消息 → 显示 Fork 按钮
  2. 点击 → 调用 C2S_ForkSession { sessionId, atMessageId }
  3. Server 调用 SDK forkSession(sessionId, { atMessageUUID })
  4. 返回新 sessionId → 自动切换到新 session
  5. 新 session 包含 fork 点之前的所有消息上下文

文件变更

文件改动
shared/protocol.ts新增 C2S_ForkSession、S2C_SessionForked
server/ws/handler.ts处理 fork 请求，调用 SDK，返回新 sessionId
web/components/chat/MessageComponent.tsxassistant 消息 hover 显示 Fork 按钮
web/hooks/useWebSocket.ts处理 session-forked 消息，切换到新 session
web/stores/sessionStore.tsfork 后刷新 session 列表
3.5 Permission Rules 持久化
复杂度: 中 改动文件: 4 SDK: PermissionUpdate + settings

问题

"Always Allow" 按钮仅在当次 session 生效。关闭后权限丢失，每次重新审批。

方案

将 AlwaysAllow 的 permission 规则持久化到 SQLite（通过 settings API），下次 query 时作为 allowedTools 传给 SDK。
