# quick-fix skill 设计

## 概述

新建 `quick-fix` skill，作为所有代码修改场景的自动触发工程纪律层。与现有 `fix` skill（重装版，手动 `/fix` 触发）互补：用户默认走 quick-fix，遇到 quick-fix 搞不定的复杂问题时手动升级到 `/fix`。

## 定位

| 属性 | quick-fix | fix |
|------|-----------|-----|
| 触发 | 自动（所有代码修改场景） | 手动（`/fix` command） |
| 执行者 | 主 Agent 直接执行 | 三 Agent 协作（研究员/开发者/测试员） |
| 适用 | 默认路径 | 快速修复搞不定时的升级路径 |

## 核心铁律

```
1. 没有证据，不能提出修复
2. 没有验证，不能声称理解
3. 没有根因，不能开始修改
4. 没有用户许可，不能扩大范围
5. 没有 checklist，不能开始执行
```

## 流程设计（6 Phase）

### Phase 0: 收集真实状态

- Read 相关代码、Grep 调用链
- 如果能运行就运行看真实报错
- 拿到事实，不凭记忆推测
- 修改代码前重新 Read 最新版本，不依赖对话早期的文件记忆

### Phase 1: 对比 + 提问

- 把真实状态与用户描述对比
- 不符 → 停下来问用户，不自己猜测、不自己做决定
- 信息模糊 → 停下来问用户
- 修 A 发现 B → 停下来问用户："修 A 发现了 B，B 应该怎么处理？"
- 完全吻合才进下一步

### Phase 2: 根因定位

- 追踪完整数据流和调用链，确认根因在哪一层
- 多假设逐一排除，不同时改多处碰运气
- 修根因不修症状（修发送方不绕接收方，修数据源不修渲染层）
- 判断错了必须明说，不悄悄换方向

### Phase 3: 方案 + checklist

- 提出修复方案：修什么、怎么修、不修什么
- 用 TodoWrite 列 checklist，每个修改点一项
- 等用户确认后才进 Phase 4

### Phase 4: 逐项执行

- 按 checklist 逐项修改，每完成一项打勾
- 不多做也不少做
- 修改前重新 Read 最新文件
- 代码级自主决定，功能级必须提问

### Phase 5: 验证

- 运行 build/lint/test
- 确认不引入新问题
- 声称完成前必须有验证输出

## 范围控制规则（贯穿全流程）

| 场景 | 行为 |
|------|------|
| 用户让修 A | 只修 A |
| 修 A 时发现 B | 问用户："发现了 B，要一起修吗？怎么修？" |
| 觉得应该加功能 | 问用户，不擅自加 |
| 判断跟之前不一致 | 明确告诉用户："我之前判断错了" |
| 代码怎么写 | 自主决定（代码级不提问） |
| 功能要不要做 | 用户决定（功能级必须提问） |

## 参考项目规则

当用户让参考另一项目时，额外激活：

1. 它怎么做，我就怎么做
2. 它有做的，我们就有做
3. 它没做的，我们就没做
4. 列 checklist 与参考项目逐项对齐
5. 需要增减必须问用户

## 自动触发词

所有涉及代码修改的场景，包括但不限于：
- bug 修复、报错、异常行为
- 新功能开发、功能实现
- 重构、优化、改进
- 参考项目对齐实现
